## 1. 设计规范

### 1.1 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| 数据库 | `music_db` | 小写+下划线 |
| 表名 | 小写单数 | `user`, `song`, `playlist` |
| 字段 | 小写蛇形 | `created_time`, `is_public` |
| 主键 | `id` | 所有表统一 BigInt |
| 外键 | 关联表_关联字段 | `user_id`, `song_id` |
| 唯一索引 | `uk_字段名` | `uk_user_email` |
| 普通索引 | `idx_字段名` | `idx_song_title` |
| 公共字段 | 所有表均包含 | `id`, `created_time`, `updated_time` |

### 1.2 公共字段

每张表都包含以下公共字段：

```sql
`id`           BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键 ID',
`created_time` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
`updated_time` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
```

### 1.3 数据类型约定

| 类型 | 使用场景 | 说明 |
|------|---------|------|
| `BIGINT` | 主键、外键 | 支持高并发场景 |
| `VARCHAR(50)` | 短文本（名称、标题） | 定长短文本 |
| `VARCHAR(255)` | 中文本（URL、邮箱） | URL 长度充足 |
| `VARCHAR(500)` | 评论内容、简介 | 200 汉字以内 |
| `TEXT` | 长文本（歌词、日志详情） | 不限长度 |
| `TINYINT(1)` | 布尔值 | 0/1 表示 |
| `INT` | 计数、序号 | `play_count`, `sort_order` |
| `DATETIME` | 时间戳 | 统一使用 |

### 1.4 存储引擎与字符集

```sql
ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
```

- **InnoDB**：支持事务、外键、行级锁
- **utf8mb4**：支持 emoji 和全部 Unicode 字符
- **unicode_ci**：基于 Unicode 的排序规则

### 核心实体关系

| 关系 | 类型 | 说明 |
|------|------|------|
| 用户 ↔ 歌单 | 1:N | 一个用户可创建多个歌单 |
| 歌单 ↔ 歌曲 | N:M | 通过 playlist_song 关联表 |
| 歌曲 ↔ 专辑 | N:1 | 一首歌属于一个专辑（允许空） |
| 歌曲 ↔ 艺人 | N:1 | 一首歌有一个主艺人 |
| 专辑 ↔ 艺人 | N:1 | 一个专辑属于一个艺人 |
| 分类 ↔ 歌曲 | N:M | 通过 song_category 关联表 |
| 用户 ↔ 收藏 | N:M(多态) | 可收藏歌曲/歌单/专辑 |
| 用户 ↔ 评论 | 1:N | 用户发表多条评论 |

## 3. 表结构详表

### 3.1 `user` — 用户表

```sql
CREATE TABLE `user` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `email`           VARCHAR(100) NOT NULL                 COMMENT '邮箱（登录账号）',
    `password`        VARCHAR(255) NOT NULL                 COMMENT 'BCrypt加密密码',
    `nickname`        VARCHAR(50)  NOT NULL                 COMMENT '昵称',
    `avatar`          VARCHAR(255) DEFAULT NULL             COMMENT '头像URL',
    `bio`             VARCHAR(500) DEFAULT NULL             COMMENT '个人简介',
    `background`      VARCHAR(255) DEFAULT NULL             COMMENT '主页背景图URL',
    `role`            VARCHAR(20)  NOT NULL DEFAULT 'USER'  COMMENT '角色: USER/VIP/ADMIN',
    `vip_expire_time` DATETIME     DEFAULT NULL             COMMENT '会员过期时间(NULL=非会员)',
    `status`          VARCHAR(20)  NOT NULL DEFAULT 'ACTIVE' COMMENT '状态: ACTIVE/BANNED',
    `created_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_email` (`email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

**范式分析**：
- ✅ 1NF：所有列不可再分，无重复组
- ✅ 2NF：主键仅 `id` 一列，无部分依赖
- ✅ 3NF：无传递依赖（role/vip_expire_time 直接依赖主键）

### 3.2 `song` — 歌曲表

```sql
CREATE TABLE `song` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `title`           VARCHAR(200) NOT NULL                 COMMENT '歌曲名',
    `artist_id`       BIGINT       NOT NULL                 COMMENT '主艺人ID',
    `album_id`        BIGINT       DEFAULT NULL             COMMENT '所属专辑ID',
    `duration`        INT          NOT NULL DEFAULT 0       COMMENT '时长(秒)',
    `audio_url`       VARCHAR(255) NOT NULL                 COMMENT '音频文件路径',
    `cover_url`       VARCHAR(255) DEFAULT NULL             COMMENT '封面图路径',
    `lyric_url`       VARCHAR(255) DEFAULT NULL             COMMENT 'LRC歌词文件路径',
    `lyrics`          TEXT         DEFAULT NULL             COMMENT '解析后的歌词JSON',
    `genre`           VARCHAR(50)  DEFAULT NULL             COMMENT '风格(流行/摇滚/古典)',
    `language`        VARCHAR(20)  DEFAULT NULL             COMMENT '语种(中文/英文/日文)',
    `release_year`    INT          DEFAULT NULL             COMMENT '发行年份',
    `status`          VARCHAR(20)  NOT NULL DEFAULT 'ACTIVE' COMMENT '状态: ACTIVE/PENDING/REJECTED',
    `uploader_id`     BIGINT       DEFAULT NULL             COMMENT '上传者(会员上传时非空)',
    `play_count`      BIGINT       NOT NULL DEFAULT 0       COMMENT '播放量',
    `created_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_song_title` (`title`),
    KEY `idx_song_artist_id` (`artist_id`),
    KEY `idx_song_album_id` (`album_id`),
    KEY `idx_song_genre` (`genre`),
    KEY `idx_song_status` (`status`),
    KEY `idx_song_play_count` (`play_count`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='歌曲表';
```

**范式分析**：
- ✅ 1NF：所有列原子化，`lyrics` 虽为 TEXT 但存储的是完整 JSON 字符串（视为一个原子值）
- ✅ 2NF：主键仅 `id`，无部分依赖
- ✅ 3NF：`artist_id`/`album_id` 是外键（引用其他表的主键），属于依赖主键而非传递依赖

### 3.6 `song_category` — 歌曲分类关联表

```sql
CREATE TABLE `song_category` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `song_id`         BIGINT       NOT NULL                 COMMENT '歌曲ID',
    `category_id`     BIGINT       NOT NULL                 COMMENT '分类ID',
    `created_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_song_category` (`song_id`, `category_id`),
    KEY `idx_sc_category_id` (`category_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='歌曲分类关联表';
```

**范式分析**：
- ✅ 1NF：原子列
- ✅ 2NF：满足。若将 `(song_id, category_id)` 作为复合主键，非 `sort_order` 列完全依赖于复合主键。此处增加自增 `id` 作为代理主键，`(song_id, category_id)` 为唯一约束，避免部分依赖
- ✅ 3NF：无传递依赖

### 3.7 `playlist` — 歌单表

```sql
CREATE TABLE `playlist` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '主键ID',

    `title`           VARCHAR(100) NOT NULL                 COMMENT '歌单标题',
    `description`     VARCHAR(500) DEFAULT NULL             COMMENT '歌单描述',
    `cover_url`       VARCHAR(255) DEFAULT NULL             COMMENT '封面图URL',
    `is_public`       TINYINT(1)   NOT NULL DEFAULT 1       COMMENT '是否公开 1=公开 0=隐藏',
    `song_count`      INT          NOT NULL DEFAULT 0       COMMENT '歌曲数量',
    `created_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_playlist_user_id` (`user_id`),
    KEY `idx_playlist_is_public` (`is_public`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='歌单表';
```

**范式分析**：
- ✅ 1NF：满足
- ✅ 2NF：满足（单列主键）
- ✅ 3NF：`song_count` 存在冗余（可以由 `playlist_song` 表 COUNT 得出），这是为了提高列表查询性能而引入的**可接受冗余**。在实际应用中通过 TRIGGER 或应用层保证一致性

### 3.11 `likes` — 喜欢表

> 注意：`likes` 而非 `like`，因为 `like` 是 MySQL 保留字。

```sql
CREATE TABLE `likes` (
    `id`              BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `user_id`         BIGINT       NOT NULL                 COMMENT '用户ID',
    `target_type`     VARCHAR(20)  NOT NULL                 COMMENT '目标类型: SONG/PLAYLIST/COMMENT',
    `target_id`       BIGINT       NOT NULL                 COMMENT '目标ID',
    `created_time`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_likes_user_target` (`user_id`, `target_type`, `target_id`),
    KEY `idx_likes_target` (`target_type`, `target_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='喜欢表';
```

**范式分析（favorite + likes）**：
- ✅ 1NF：原子列
- ✅ 2NF：使用自增代理主键，无复合主键部分依赖问题
- ✅ 3NF：无传递依赖
- 💡 设计说明：`target_type` + `target_id` 是多态关联（Polymorphic Association）设计。不需要为每种目标类型建独立表，减少了表数量且易于扩展

## 6. 范式分析

### 6.1 第一范式（1NF）—— 列原子性

**规则**：每列都是不可再分的原子值，无重复组。

| 表 | 检查 | 结论 |
|----|------|------|
| `user` | 所有列均为标量值 | ✅ |
| `song` | `lyrics` 存储 JSON 字符串（视为原子值） | ✅ |
| `playlist_song` | 无数组，无重复组 | ✅ |
| `favorite` | `target_type` + `target_id` 虽有多态但存储为独立标量列 | ✅ |

> 唯一需要注意的：`lyrics` 列存储的 JSON 字符串。从数据库层面它是一个完整的 TINYTEXT/TEXT 值，应用层解析其结构但不影响 1NF。

### 6.2 第二范式（2NF）—— 完全函数依赖

**规则**：所有非主键列必须完全依赖于主键（对复合主键尤为重要）。

| 表 | 主键类型 | 检查 | 结论 |
|----|---------|------|------|
| 所有单列主键的表 | 单列(ID) | 不可能有部分依赖 | ✅ |
| `playlist_song` | 自增 ID (代理) | 非主键列不依赖复合键部分 | ✅ |
| `song_category` | 自增 ID (代理) | 同上 | ✅ |
| `favorite` | 自增 ID (代理) | 同上 | ✅ |

> **设计决策**：所有多对多关联表均使用自增 ID 作为代理主键，并用 UNIQUE 约束替代复合主键。这样：
> 1. 消除了部分函数依赖的可能性
> 2. 子表引用关联表时只需一个字段
> 3. 避免 InnoDB 聚簇索引过大导致二级索引膨胀

### 6.3 第三范式（3NF）—— 无传递依赖

**规则**：非主键列不能依赖于其他非主键列。

| 场景 | 是否存在传递依赖 | 结论 |
|------|-----------------|------|
| `song` 表中 `artist_id` → 通过外键关联 `artist` 表 | 外键列属于直接依赖主键，不属于传递依赖 | ✅ |
| `playlist` 表中 `song_count` | 可由 `playlist_song` 的 COUNT 计算得出，但这是**刻意冗余**（性能考虑） | ⚠️ 可接受 |
| `user` 表中 `status` / `role` | 直接依赖用户主键 | ✅ |
| `report` 表中 `handled_time` | 直接依赖报告主键，且在 RESOLVED 时才有值 | ✅ |

> **注意**：
> - `playlist.song_count` 是 3NF 的刻意违反（反范式化），目的是避免每次列表查询都做 COUNT 子查询
> - 一致性保证：添加/删除歌单歌曲时，通过应用层或定期任务同步更新 `song_count`
> - 类似的冗余还可考虑：`song.play_count`（替代实时 COUNT `play_history`）

### 6.4 范式总结

| 表 | 满足范式 | 反范式列（如有） |
|----|---------|-----------------|
| user | 3NF | — |
| song | 3NF | — |
| album | 3NF | — |
| artist | 3NF | — |
| category | 3NF | — |
| song_category | 3NF | — |
| playlist | 3NF | `song_count`（冗余计数） |
| playlist_song | 3NF | — |
| comment | 3NF | — |
| favorite | 3NF | — |
| likes | 3NF | — |
| recharge_record | 3NF | — |
| room_message | 3NF | — |
| notification | 3NF | — |
| report | 3NF | — |
| banner | 3NF | — |
| system_notice | 3NF | — |

### 7.2 Redis 补充存储（非 MySQL）

以下数据仅存储在 Redis 中，不落 MySQL：

| 数据 | Redis 结构 | TTL | 说明 |
|------|-----------|-----|------|
| Token 黑名单 | String `bl_token:{jti}` | 等于 Token 剩余 | 登出后失效 |
| 热门歌曲缓存 | String `cache:hot_songs` | 300s | 减少 DB 查询 |
| 歌词缓存 | String `cache:lyrics:{songId}` | 1800s | LRC 解析结果 |
| 歌房状态 | Hash `room:{roomId}` | — | 实时状态 |
| 歌房成员 | Set `room:{roomId}:members` | — | 在线成员 |
| API 限流 | String `ratelimit:{userId}:{uri}` | 60s | 滑动窗口 |
| 在线状态 | String `online:{userId}` | 300s | 心跳续期 |
| 播放计数 | String `play_count:buffer:{songId}` | 临时 | 定时刷入 DB |
| 播放进度 | String `progress:{userId}:{songId}` | 7天 | 进度记忆 |

## 附录

### A. 范式检查清单

```sql
-- 1NF 检查：所有查询结果是否都是标量值？
-- 每张表均满足，无数组、JSON 嵌套仅 storage 层，不影响原子性。

-- 2NF 检查：是否存在仅依赖于复合主键部分的列？
-- 所有多对多表均使用代理主键 + UNIQUE 约束，没有复合主键。

-- 3NF 检查：是否存在依赖于非主键列的列？
-- playlist.song_count 是唯一刻意违反，属于可接受的性能优化。
```
