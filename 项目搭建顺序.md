## 1. 前言与依赖关系总览

### 1.1 模块依赖拓扑

```
                    ┌──────────────┐
                    │   基础设施    │
                    │  MySQL/Redis │
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │  后端核心骨架  │
                    │ 统一响应/异常 │
                    │  处理/跨域    │
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   用户模块    │ ◄──── 所有业务模块的前提
                    │ 注册/登录/JWT │
                    └──────┬───────┘
                           ▼
              ┌────────────┴────────────┐
              │                         │
     ┌────────┴────────┐      ┌─────────┴────────┐
     │   音乐管理模块    │      │     文件模块      │
     │ 歌曲/专辑/艺人    │◄────►   上传/下载       │
     └────────┬─────────┘      └──────────────────┘
              │
     ┌────────┴────────┐      ┌──────────────────┐
     │   交互模块       │      │   音乐播放模块     │
     │ 评论/收藏/歌单   │◄────►   播放器/歌词      │
     └────────┬─────────┘      └──────────────────┘
              │
     ┌────────┴────────┐      ┌──────────────────┐
     │   通知/举报模块   │      │   歌房模块        │
     │  推送/处理       │      │  WebSocket/PubSub│
     └────────┬─────────┘      └──────────────────┘
              │
     ┌────────┴────────┐
     │  管理员后台模块   │
     │  看板/审核/运营   │
     └─────────────────┘
```

### 1.2 搭建原则

1. **从底层到上层**：基础设施 → 核心功能 → 增值功能 → 运营功能
2. **模块独立可运行**：每个阶段完成后该模块可独立测试
3. **前后端同步**：每个模块优先完成后端 API，前端页面紧随其后
4. **可并行阶段**：部分无依赖的模块可并行开发

### 5.2 我已完成的关键设计决策

- **统一响应体** `R<T>`：`{ code, message, data, timestamp }`
- **全局异常处理**：区分业务异常（BusinessException）和系统异常，统一返回格式
- **错误码体系**：1xxx 用户模块 / 2xxx 音乐 / 3xxx 歌房 / 4xxx 文件 / 5xxx 会员 / 9xxx 通用
- **ThreadLocal 请求上下文**：拦截器解析 JWT 后注入 userId，业务层直接获取
- **Redis Key 规范**：统一前缀 `cache:`, `room:`, `ratelimit:`, `bl_token:` 等
- **限流方案**：基于 Redis + Lua 原子脚本的滑动窗口

### 13.3 WebSocket 消息协议

```json
{
  "type": "CHAT | SYNC | SYSTEM | VOICE | HEARTBEAT",
  "userId": 1,
  "nickname": "用户昵称",
  "payload": {},
  "timestamp": 1718000000000
}
```

| 消息类型 | 方向 | 说明 |
|---------|------|------|
| HEARTBEAT | 客户端 → 服务端 → 客户端 | 心跳（30s 间隔），服务端回复 HEARTBEAT pong |
| CHAT | 客户端 → Redis → 房间广播 | 聊天消息，payload: {content: "xxx"} |
| SYNC | 客户端 → Redis → 房间广播（排除发送者） | 播放同步，payload: {isPlaying, songId, progress} |
| SYSTEM | 服务端 → 房间广播 | 系统消息（加入/离开/踢出/移交），自动可见 |
| VOICE | 客户端 → Redis → 房间广播 | 语音消息，payload: {url, duration} |

### 13.4 Redis 数据结构

| Key | 类型 | 说明 |
|-----|------|------|
| room:{roomId} | Hash | 房间状态（name/ownerId/status/isPublic/password/currentSongId/progress/isPlaying/playMode/maxMembers/createdAt/memberCount） |
| room:{roomId}:members | Set | 在线成员（userId 集合） |
| room:{roomId}:queue | String | 播放队列 JSON [{songId, addedBy}] |

### 13.5 核心同步流程

```
房主操作 → WebSocket (type=SYNC) → RoomWebSocketHandler.handleSync()
  → 更新 Redis Hash room:{roomId}
  → RedisMessagePublisher.publish(channel:room:sync, message)
  → RoomMessageSubscriber.handleRoomSync() — 所有服务实例收到
  → RoomSessionManager.broadcastToRoom() — 遍历房间内所有 WebSocket 会话
  → 广播 SYNC 消息给其他成员（排除发送者）
```

### 13.6 关键实现逻辑

- **WebSocket 鉴权**：握手拦截器从 URL 参数取 token → JWT 验证 → userId 注入 session 属性
- **自动解散**：全员离开房间时自动清理 Redis 资源
- **房主自动移交**：房主离开时自动将房主移交给房间内下一成员
- **跨实例广播**：WebSocket Handler 发布到 Redis Channel → 所有实例的 RoomMessageSubscriber 接收 → 本地 WebSocket 广播
- **会话管理**：RoomSessionManager 维护 sessionId→roomId→userId 三级映射，支持多端登录
- **TTL 过期**：房间 12 小时无活跃自动过期，避免僵尸房间
