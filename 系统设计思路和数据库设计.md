2.1 系统设计思路

本系统采用B/S（Browser/Server）架构，前后端分离的开发模式。前端负责用户界面的渲染与交互逻辑，后端提供RESTful API接口处理业务逻辑与数据存取。前后端通过HTTP协议进行通信，数据交换格式采用JSON。

系统整体分为表现层（前端）、业务逻辑层（后端Service）、数据访问层（后端DAO/Mapper）和数据库层四个层次。表现层负责接收用户操作并展示数据；业务逻辑层封装核心业务规则与流程；数据访问层通过MyBatis框架与数据库进行交互，完成数据的持久化操作。这种分层设计降低了各层之间的耦合度，提高了代码的可维护性和可测试性。

在技术选型方面，后端选择Spring Boot框架，其自动配置和起步依赖特性能够显著提高开发效率；持久层采用MyBatis框架，便于编写复杂的SQL查询并实现灵活的ORM映射；数据库选用MySQL 8.0，支持事务、索引等特性，能够满足音乐数据存储与检索的需求；前端采用Vue.js框架结合Element Plus组件库，实现响应式用户界面和流畅的交互体验。

在安全设计方面，系统采用JWT（JSON Web Token）实现无状态的用户身份认证。用户登录成功后，服务端生成包含用户身份信息的JWT Token并返回给客户端，客户端在后续请求中携带该Token，服务端通过拦截器验证Token的有效性，从而确保接口访问的合法性与安全性。

### 2.2.4 缓存数据结构设计

为提升系统高并发场景下的响应性能，降低 MySQL 数据库的查询压力，本系统引入 Redis 作为缓存层。以下数据仅存储在 Redis 中，不落 MySQL 持久化存储，主要用于热点数据缓存、实时状态维护和临时计数等场景。各缓存数据的设计结构如表 2-3 所示。

表 2-3 Redis 缓存数据结构

| 数据         | Key 格式                   | 数据结构       | TTL                      | 说明                                                         |
| ------------ | -------------------------- | -------------- | ------------------------ | ------------------------------------------------------------ |
| Token 黑名单 | bl_token:{jti}             | String         | 等于 Token 剩余有效期    | 用户登出后将 Token 加入黑名单，使其立即失效                  |
| 热门歌曲缓存 | cache:hot_songs            | String（JSON） | 300s                     | 缓存首页热门歌曲列表，每5分钟刷新，减少 DB 查询压力          |
| 歌词缓存     | cache:lyrics:{songId}      | String（JSON） | 1800s                    | 缓存解析后的 LRC 歌词 JSON 数据，避免重复解析文件            |
| 歌房状态     | room:{roomId}              | Hash           | 无固定 TTL（随房间销毁） | 存储歌房实时状态，包括房主 ID、当前播放歌曲、播放进度、房间创建时间等字段 |
| 歌房成员     | room:{roomId}:members      | Set            | 无固定 TTL（随房间销毁） | 存储歌房当前在线成员的用户 ID 集合                           |
| API 限流     | ratelimit:{userId}:{uri}   | String（计数） | 60s                      | 基于滑动窗口实现接口限流，记录用户在指定接口的请求次数       |
| 在线状态     | online:{userId}            | String         | 300s                     | 通过心跳机制维护用户在线状态，每次请求续期 5 分钟            |
| 播放进度     | progress:{userId}:{songId} | String（JSON） | 7天                      | 存储用户对每首歌曲的播放进度（秒数），支持断点续播           |

通过上述 Redis 缓存设计，系统在热门歌曲榜单加载、歌词展示、歌房实时通信等高频访问场景中能够显著提升响应速度，同时有效保护后端数据库的稳定性。
