# 🎵 歌房模块设计文档

## 1. 概述

歌房（Room）是一个**实时同步听歌**的多人协作空间。用户在同一个歌房中，可以同步收听同一首歌，实时聊天，共同构建播放队列。歌房的所有状态实时同步，所有成员共享听歌体验。

### 1.1 核心目标

- 多人实时同步听歌（房主操控播放，所有成员同步收听）
- 实时文字聊天（房间内所有成员可见）
- 灵活的歌曲来源（歌单、专辑、收藏、艺人歌曲）
- 简易的房主管理机制（创建、踢人、转交、解散）

### 1.2 角色权限

| 角色 | 创建歌房 | 播放控制 | 管理成员 | 解散房间 | 加密房间 |
|------|---------|---------|---------|---------|---------|
| 普通用户 | ❌ | ❌ | ❌ | ❌ | ❌ |
| VIP 会员 | ✅ | ✅（仅当自己是房主） | ✅（仅当自己是房主） | ✅（仅当自己是房主） | ✅ |
| 管理员 | ✅ | ✅（仅当自己是房主） | ✅（仅当自己是房主） | ✅（仅当自己是房主） | ✅ |

## 2. 数据模型

### 2.1 Redis 存储（房间实时状态）

房间状态**全部存储在 Redis** 中，无 MySQL 持久化。Redis 在内存中操作保证实时性。

```
room:{roomId}                  → Hash    房间元数据
room:{roomId}:members          → Set     在线成员 ID 集合
room:{roomId}:queue            → String  播放队列 (JSON 数组)
room:{roomId}:joinOrder        → List    加入顺序列表（用于房主转移）
```

#### 房间 Hash 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | String | 房间名称 |
| `ownerId` | String(数字) | 房主用户 ID |
| `status` | String | WAITING / PLAYING / PAUSED |
| `isPublic` | String | "1"=公开, "0"=私密 |
| `password` | String | 房间密码（私密时有效，空=无密码） |
| `currentSongId` | String(数字/空) | 当前播放歌曲 ID |
| `progress` | String(数字) | 播放进度（毫秒） |
| `isPlaying` | String | "1"=播放中, "0"=暂停 |
| `playMode` | String | 播放模式：SEQUENCE（列表循环） |
| `maxMembers` | String(数字) | 最大成员数（2/4/8/16） |
| `createdAt` | String(时间戳) | 创建时间 |

#### TTL

- 房间 Hash 和队列：**12 小时**无活跃自动过期
- 成员 Set 和加入顺序 List：**12 小时**

### 2.2 MySQL 持久化（消息记录）

#### room_message 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT (PK) | 主键 |
| `room_id` | BIGINT | 房间 ID |
| `user_id` | BIGINT | 发送者 ID（0=系统消息） |
| `type` | VARCHAR | TEXT / VOICE / SYSTEM |
| `content` | TEXT | 消息内容（文本或语音 URL） |
| `created_time` | DATETIME | 创建时间 |

> 仅保留消息记录供新成员查看历史（默认加载最近 50 条）。房间解散后消息记录仍保留在数据库中。

### 2.3 VO（视图对象）

```
RoomVO:
  id, name, ownerId, ownerNickname,
  status, isPublic, hasPassword,
  currentSongId, currentSongTitle, currentSongCover, currentSongArtist,
  progress, isPlaying, playMode,
  memberCount, maxMembers, createdAt

RoomMemberVO:
  userId, nickname, avatar, isOwner, isOnline, isVip

RoomMessageVO:
  id, roomId, userId, nickname, type, content, createdAt
```

## 3. API 设计

所有 REST API 路径前缀：`/api/rooms`

### 3.1 房间管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/` | 创建歌房 | **VIP/管理员** |
| GET | `/` | 获取公开房间列表 | 任意 |
| GET | `/{id}` | 获取房间详情 | 任意 |

**创建房间请求体 (CreateRoomDTO)**:
```json
{
  "name": "房间名称",       // 必填，最长 50 字符
  "isPublic": true,         // true=公开 false=私密
  "password": "1234",       // 私密时必填，最长 20 字符
  "maxMembers": 8           // 2/4/8/16
}
```

### 3.2 成员管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/{id}/join` | 加入房间（私密需密码） | 已登录 |
| POST | `/{id}/leave` | 离开房间 | 已登录 |
| POST | `/{id}/kick/{targetId}` | 踢出成员 | 房主 |
| POST | `/{id}/transfer/{targetId}` | 移交房主 | 房主 |
| POST | `/{id}/dismiss` | 解散房间 | 房主 |

### 3.3 队列管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/{id}/queue` | 获取播放队列 | 已登录 |
| POST | `/{id}/queue` | 添加单曲到队列 | 房主 |
| DELETE | `/{id}/queue` | 从队列移除歌曲 | 房主 |
| POST | `/{id}/queue/playlist/{playlistId}` | 加载歌单所有歌曲 | 房主 |
| POST | `/{id}/queue/album/{albumId}` | 加载专辑所有歌曲 | 房主 |
| POST | `/{id}/queue/favorites` | 加载收藏歌曲 | 房主 |

### 3.4 消息查询

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/{id}/members` | 获取在线成员列表 | 已登录 |
| GET | `/{id}/messages?limit=50` | 获取最近消息 | 已登录 |

## 4. WebSocket 协议

连接地址：`ws://host/ws/room/{roomId}?token={jwt}`

### 4.1 消息格式

所有消息统一为 JSON 格式：

```json
{
  "type": "消息类型",
  "userId": 1,
  "nickname": "用户昵称",
  "payload": { ... },
  "timestamp": 1704067200000
}
```

### 4.2 消息类型

| 类型 | 方向 | 说明 |
|------|------|------|
| `HEARTBEAT` | 双向 | 心跳检测，客户端 30 秒发送一次 |
| `CHAT` | 客户端→服务端→房间其他人 | 聊天消息 |
| `SYNC` | 房主→服务端→房间其他人 | 播放同步 |
| `SYSTEM` | 服务端→房间所有人 | 系统消息（成员加入/离开等） |
| `VOICE` | 客户端→服务端→房间其他人 | 语音消息 |

### 4.3 消息明细

**SYNC（播放同步，仅房主可发送）**:
```json
{
  "type": "SYNC",
  "payload": {
    "roomId": 1,
    "songId": 100,
    "isPlaying": true,
    "progress": 45000,
    "action": "PLAY"
  }
}
```

**CHAT**:
```json
{
  "type": "CHAT",
  "payload": {
    "content": "这首歌真好听！"
  }
}
```

### 4.4 消息分发架构

```
客户端 → RoomWebSocketHandler → Redis Pub/Sub
                                   ↓
                          RoomMessageSubscriber
                                   ↓
                      RoomSessionManager.broadcastToRoom()
                                   ↓
                              其他客户端（排除发送者）
```

这种通过 Redis Pub/Sub 的架构支持**多后端实例部署**——跨实例的成员也能收到消息。

## 6. 业务流程

### 6.1 创建房间

```
用户( VIP/管理员 ) → 点击「创建歌房」
  → 填写表单（名称/公开/密码/人数）
  → POST /api/rooms
  → 后端校验角色（非 VIP/管理员 拒绝）
  → Redis 创建房间 Hash
  → 返回 RoomVO（含 roomId）
  → 前端 router.push(`/rooms/${roomId}`)
```

### 6.2 加入/离开房间

```
加入:
  用户 → 点击房间卡片（私密房间需输入密码）
  → POST /api/rooms/{id}/join (含 password)
  → 后端校验密码/人数上限
  → 加入 Redis Set → 发送系统消息 → WebSocket 广播成员变更
  → 前端进入详情页 → 建立 WebSocket 连接

离开:
  用户 → 点击「← 退出房间」
  → 断开 WebSocket
  → 销毁本地音频
  → POST /api/rooms/{id}/leave
  → 如果是房主离开 → 自动解散房间（所有成员收到 SYSTEM 消息 → 跳转回列表）
  → 如果是普通成员离开 → 从成员集合移除 → WebSocket 广播
  → 前端 router.push('/rooms')
```

**房主离开的完整流程**：
1. 房主点击退出 → POST `/api/rooms/{id}/leave`
2. 后端检测到离开者是房主 → 删除 Redis 中所有房间数据
3. 后端通过 WebSocket 向房间内所有成员广播 `SYSTEM` 消息（"房间已被解散"）
4. 所有成员的前端收到后自动跳转 `/rooms` 列表页
5. 房间音频停止、WebSocket 断开

### 6.3 播放控制

```
房主操作:

  添加歌曲:
    房主点击「歌单/专辑/收藏」Tab
    → 从对应 API 获取列表
    → 点击「+ 队列」单独添加 或「全部添加到队列」
    → 后端将歌曲 ID 加入队列 JSON 数组

  播放歌曲:
    房主在队列中点击某首歌曲的 ▶ 按钮
    → 获取歌曲完整信息（audioUrl）
    → 本地创建 Audio 实例开始播放
    → 发送 WebSocket SYNC 消息（songId, progress=0, isPlaying=true）
    → 其他成员收到 SYNC → 也创建 Audio 实例播放同一首歌
    
  播放控制:
    ⏸ 暂停 → 暂停本地 Audio → 发送 SYNC (isPlaying=false, progress)
    ⏭ 下一首 → 停止当前 → 获取下一首歌 → 开始播放 → 发送 SYNC
    
  进度同步:
    timeupdate 事件中，每 10 秒同步一次进度（仅房主 → 其他成员）
    其他成员收到 progress 后 seek 到对应位置

  歌曲结束:
    ended 事件 → 房主自动播放下 → 发送 SYNC
```

### 6.4 实时聊天

```
用户输入消息 → 点击发送/回车
  → 本地立即显示消息（优化体验）
  → 通过 WebSocket 发送 CHAT 消息
  → 后端保存到 MySQL room_message 表
  → 通过 Redis Pub/Sub 广播给房间其他成员（排除发送者）
  → 其他成员收到 CHAT 消息 → 追加到聊天列表
```

### 6.5 解散房间

```
房主 → 点击「解散房间」→ 确认弹窗
  → POST /api/rooms/{id}/dismiss
  → 后端删除 Redis 中所有房间数据
  → 前端跳转 /rooms
  → 其他成员通过 WebSocket 收到 CLOSE 消息（连接断开）
  → 其他成员前端跳转 /rooms
```

## 7. 与当前实现的差异点（待修改）

当前代码已有较完整的歌房实现，但与需求存在以下行为差异，需修改：

### 7.1 ⚠️ 创建权限限制

| 当前行为 | 目标行为 |
|---------|---------|
| 所有已登录用户均可创建歌房 | **仅 VIP 会员和管理员**可创建歌房 |

**修改位置**：`RoomServiceImpl.createRoom()` — 增加角色校验

### 7.2 ⚠️ 房主离开行为

| 当前行为 | 目标行为 |
|---------|---------|
| 房主离开时：有 VIP 成员则自动移交房主，无则关闭 | **房主离开 → 一律解散房间**，所有成员跳转回列表 |

**修改位置**：`RoomServiceImpl.leaveRoom()` — 移除 VIP 自动接管逻辑，统一解散

### 7.3 ⚠️ 房间列表自动刷新

| 当前行为 | 目标行为 |
|---------|---------|
| `RoomList.vue` 仅在 `onMounted` 时加载一次 | **每 5 秒轮询**，自动筛除已解散的房间 |

**修改位置**：`RoomList.vue` — 添加 `setInterval` 定时刷新

### 7.4 房间详情页 WebSocket 断开处理

房主离开导致房间解散时，其他成员的 WebSocket 连接会断开。需要在前端监听 `onclose` 事件，自动跳转回 `/rooms`。

**修改位置**：`RoomDetail.vue` — WebSocket `onclose` 回调中检测房间不存在，自动导航到列表页

### 8.1 RoomList.vue 关键逻辑

- `onMounted()` + `setInterval(5000)` 定时刷新房间列表
- 创建按钮仅对 `authStore.isVip || authStore.isAdmin` 可见
- 自动筛除已解散的房间（API 返回时已过滤，轮询保证列表实时性）

### 8.2 RoomDetail.vue 关键逻辑

- **音频管理**：独立的 `Audio` 实例（独立于全局播放器），生命周期绑定房间
- **播放同步**：房主导播，成员同步。房主通过 WebSocket SYNC 分发播放状态
- **退出清理**：断开 WebSocket → 销毁 Audio → 调用 leave API
- **房间解散（被动）**：WebSocket 断开 → 检测房间不存在 → 自动跳转列表
- **音乐来源切换**：歌单 / 专辑 / 收藏 / 艺人四个 Tab 切换

## 10. 错误码

| 错误码 | 说明 | 场景 |
|--------|------|------|
| `ROOM_NOT_FOUND` | 房间不存在 | 房间已解散或 ID 无效 |
| `ROOM_FULL` | 房间已满 | 成员数达到 maxMembers |
| `ROOM_PASSWORD_WRONG` | 密码错误 | 私密房间密码不匹配 |
| `ROOM_ALREADY_JOINED` | 已在房间中 | 重复加入 |
| `ROOM_NOT_OWNER` | 非房主操作 | 非房主尝试踢人/解散等 |
| `ROOM_PERMISSION_DENIED` | 无权限创建 | 非 VIP/管理员尝试创建房间 |

## 11. 安全考虑

1. **JWT 鉴权**：WebSocket 连接时通过 URL token 参数鉴权（`RoomWebSocketInterceptor`）
2. **房主校验**：所有写操作（踢人、解散、播放控制等）需校验操作人是否为房主
3. **密码保护**：私密房间的密码存储在 Redis Hash 中，API 返回的 RoomVO 中 `hasPassword` 仅表示是否有密码，不暴露密码本身
4. **消息限制**：聊天内容最长 500 字符，WebSocket 帧大小限制 64KB
5. **创建权限**：仅 VIP/管理员可创建房间（新增限制）
6. **Redis TTL**：房间 12 小时无活跃自动过期，防止僵尸房间
