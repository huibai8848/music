## 2. 用户与认证模块

### 2.1 功能概述

用户注册、登录、鉴权、个人信息管理、会员体系。是整个平台的认证与授权基础。

### 2.2 实体设计

```
User 实体
├── id: Long (PK)
├── email: String (UNIQUE)       // 登录账号，邮箱格式
├── password: String              // BCrypt 加密密文
├── nickname: String              // 昵称
├── avatar: String                // 头像 URL
├── bio: String                   // 个人简介
├── background: String            // 个人主页背景图 URL
├── role: Enum(USER, VIP, ADMIN) // 角色
├── vipExpireTime: LocalDateTime  // 会员过期时间（null 表示非会员）
├── status: Enum(ACTIVE, BANNED) // 状态
├── createdTime: LocalDateTime
└── updatedTime: LocalDateTime
```

### 2.3 接口方法

| 方法 | URL | 说明 | 鉴权 | 限流 |
|------|-----|------|------|------|
| `getPublicKey` | GET /api/auth/public-key | 获取 RSA 公钥 | 无 | — |
| `register` | POST /api/auth/register | 邮箱+密码+昵称注册 | 无 | 5/min/IP |
| `login` | POST /api/auth/login | 登录，支持 RSA 加密密码 | 无 | 10/min/IP |
| `refreshToken` | POST /api/auth/refresh | 刷新 access_token | RefreshToken | — |
| `logout` | POST /api/auth/logout | 登出，拉黑 Token | JWT | — |
| `forgotPassword` | POST /api/auth/forgot-password | 发送密码重置验证码 | 无 | 3/300s |
| `resetPassword` | POST /api/auth/reset-password | 验证码重置密码 | 无 | — |
| `getCurrentUser` | GET /api/users/me | 获取当前用户信息 | JWT | — |
| `updateProfile` | PUT /api/users/me | 修改昵称/头像/简介/背景 | JWT | — |
| `changePassword` | PUT /api/users/me/password | 旧密码验证修改 | JWT | — |
| `rechargeMembership` | POST /api/users/me/membership | 充值会员 | JWT | — |
| `getPublicProfile` | GET /api/users/{id} | 查看他人公开主页 | 可选 JWT | — |

### 2.4 核心流程

#### 2.4.1 注册流程

```
① 校验 email 格式 + 密码强度（6-20位）
② BCrypt.hashpw(password) 加密密码
③ 检查 email 是否已注册（uk_user_email 唯一约束）
④ INSERT user 记录（默认 role=USER, status=ACTIVE）
⑤ 自动登录，返回 JWT tokens + 用户信息
```

#### 2.4.2 登录流程

```
① 根据 email 查询用户
② BCrypt.checkpw(password, user.password) 校验密码
③ 校验用户状态：BANNED 则禁止登录
④ 生成 JWT：access_token（2h 过期，含 userId/role/jti）
⑤ 生成 refresh_token（7d 过期），存入 Redis
⑥ 返回 tokens + 用户信息
```

**RSA 加密传输**：前端可通过 jsencrypt 库对密码进行 RSA 加密后传输，后端 RsaUtil 自动解密。先调用 `GET /api/auth/public-key` 获取公钥。

#### 2.4.3 Token 刷新

```
① 校验 refresh_token 有效性
② 检查 Redis 中是否存在该 refresh_token（是否已被吊销）
③ 生成新的 access_token（旧的不主动失效，由前端管理）
④ 返回新 access_token
```

#### 2.4.4 登出

```
① 解析当前 access_token 的 jti
② 将 jti 加入 Redis 黑名单，TTL = token 剩余有效期
③ 删除 refresh_token（Redis）
④ 返回成功
```

### 2.5 JWT 拦截器鉴权规则

**JwtInterceptor** 拦截所有 `/api/**` 请求：

| 规则 | 说明 |
|------|------|
| OPTIONS 预检 | 直接放行 |
| 静态资源 `/data/**`、`/error` | 直接放行 |
| 上传文件访问 `/api/files/**`(不含 upload) | 直接放行 |
| 精确路径白名单（登录/注册/公开接口） | 所有方法放行 |
| GET 请求白名单前缀（songs/albums/artists等） | GET 放行，写操作需认证 |
| 管理员前缀 `/api/admin` | 额外校验 role=ADMIN |
| 其他接口 | 需要有效 access_token |

**白名单可选的用户上下文注入**：白名单 GET 请求尝试解析 Token 并注入用户信息，解析失败不阻止请求。

### 2.6 边界条件

| 边界 | 处理方式 |
|------|---------|
| 重复注册同一邮箱 | 返回 1001(USER_ALREADY_EXISTS) |
| Token 过期 | 返回 401，前端自动 refresh_token |
| Token 在黑名单中 | 返回 401，要求重新登录 |
| 被封禁用户登录 | 返回 1004(USER_BANNED) |
| 会员已过期 | 正常登录，上传等功能受限 |
| refresh_token 过期 | 返回 401，前端跳转登录页 |

## 3. 音乐管理模块

### 3.1 功能概述

歌曲、专辑、艺人的 CRUD 管理，模糊搜索，多维度筛选，播放量统计，歌曲审核流程。会员可上传音乐待审核。

### 3.2 实体设计

```
Song 实体
├── id: Long (PK)
├── title: String                // 歌曲名
├── artistId: Long (FK)          // 主艺人
├── albumId: Long (FK, nullable) // 所属专辑
├── duration: Integer             // 时长（秒）
├── audioUrl: String              // 音频文件路径
├── coverUrl: String              // 封面图路径
├── lyricUrl: String              // LRC 歌词路径（原始文件）
├── lyrics: TEXT                  // 解析后的歌词 JSON
├── genre: String                 // 风格（流行/摇滚/古典等）
├── language: String              // 语种
├── releaseYear: Integer          // 发行年份
├── status: Enum(PENDING, ACTIVE, REJECTED) // 审核状态
├── uploaderId: Long (FK)        // 上传者（会员上传时非空）
├── playCount: Long               // 播放量
├── categoryIds: List<Long>       // 非数据库字段，分类 ID 列表
├── createdTime: LocalDateTime
└── updatedTime: LocalDateTime

Album 实体
├── id: Long (PK)
├── title: String                 // 专辑名
├── artistId: Long (FK)           // 所属艺人
├── coverUrl: String              // 封面
├── description: TEXT             // 简介
├── releaseDate: LocalDate        // 发行日期
├── createdTime: LocalDateTime
└── updatedTime: LocalDateTime

Artist 实体
├── id: Long (PK)
├── name: String                  // 艺人名
├── avatar: String                // 头像
├── bio: TEXT                     // 简介
├── country: String               // 国籍/地区
├── createdTime: LocalDateTime
└── updatedTime: LocalDateTime
```

### 3.4 核心流程

#### 3.4.1 搜索逻辑

```
① 接收 q 参数（歌曲名/艺人名/专辑名）
② SQL LIKE %q% 模糊匹配三张表（LEFT JOIN artist 和 album）
③ 按相关性排序：标题匹配 > 艺人匹配 > 专辑匹配
④ 仅搜索 status=ACTIVE 的歌曲
⑤ 分页返回（默认 20 条/页）
```

#### 3.4.2 播放量统计

```
① 前端播放歌曲 → POST /api/history { songId } （记录播放历史）
② 后端 PlayHistoryService.recordPlay() 执行：
   - INSERT INTO play_history (user_id, song_id, played_time) VALUES (..., NOW())
   - UPDATE song SET play_count = play_count + 1 WHERE id = {songId}
③ 排行榜/热门歌曲直接从 song.play_count 字段查询（ORDER BY play_count DESC）
④ 播放量实时准确，无需定时任务刷入
```

#### 3.4.3 会员上传流程

```
① 校验会员身份（role=VIP 或 ADMIN），检查会员过期时间
② 校验每日上传次数（Redis 计数，默认 5 次/天）
③ 校验会员存储空间（Redis 追踪，VIP 上限 500MB）
④ 上传音频/封面/歌词文件 → 写入磁盘 → 获取 URL
⑤ 使用 Apache Tika 从上传文件提取音频时长
⑥ 解析 songInfo JSON，构建 Artist/Album/Song 实体
⑦ 自动映射 genre/language/releaseYear → category 分类 ID
⑧ 事务性写入数据库（艺人/专辑/歌曲原子提交）
⑨ 成功后递增上传计数 + 更新存储使用量
⑩ 管理员上传直接 ACTIVE，会员上传需审核（PENDING）
```

### 3.5 边界条件

| 边界 | 处理方式 |
|------|---------|
| 搜索无结果 | 返回空列表 |
| 歌曲不存在 | 400, code=2001 |
| 专辑/艺人不存在 | 400, code=2003/2004 |
| 会员上传非 MP3/非图片 | 400, code=4001 |
| 非会员调用上传 | 400, code=5002 |
| 上传次数超限 | 400, code=5003 |
| 存储空间不足 | 400, code=5004 |
| 上传失败消耗次数 | 次数递增在成功后执行，失败不消耗 |

## 4. 音乐播放模块

### 4.1 功能概述

全局音频播放器，LRC 歌词同步，播放队列管理，4 种播放模式，播放进度记忆，最近播放记录。

### 4.2 数据结构

```
// 后端无独立实体，使用 Redis 存储状态
Redis:
├── cache:lyrics:{songId} → JSON [{time, text}]       // 歌词缓存
└── progress:{userId}:{songId} → String "45000"       // 播放进度（ms）

// 前端 Pinia Store 状态
PlaybackState:
├── currentSong: Song | null       // 当前歌曲
├── queue: Song[]                  // 播放队列
├── queueIndex: number             // 队列当前位置
├── mode: PlayMode                 // SEQUENCE | REPEAT_ONE | SHUFFLE | LOOP
├── volume: number                 // 音量 0-1
├── progress: number               // 当前进度 ms
├── duration: number               // 总时长 ms
├── isPlaying: boolean
└── isMuted: boolean
```

### 4.3 接口方法

| 方法 | URL | 说明 | 鉴权 |
|------|-----|------|------|
| GET | /api/history | 最近播放记录（最多 100 条） | JWT |
| POST | /api/history | 记录播放历史 | JWT |
| DELETE | /api/history | 清除播放记录 | JWT |
| POST | /api/progress | 上报播放进度（存 Redis，TTL 7天） | JWT |
| GET | /api/progress/{songId} | 获取播放进度 | JWT |

### 4.4 核心流程

#### 4.4.1 全局播放器（前端）

```
① Pinia 状态管理 playbackStore
② Audio 实例全局单例，不随页面切换销毁
③ 组件结构：
   PlayerBar.vue      — 底部固定播放条（歌曲信息/播放控制/进度条/音量）
   DiscAnimation.vue  — 黑胶碟片旋转动画
   LyricsPanel.vue    — 歌词面板（展开时）
   PlayQueue.vue      — 播放队列抽屉
④ 跨页面保持播放状态（切换页面不中断音乐）
```

#### 4.4.2 LRC 歌词同步

```
后端：
① 读取 LRC 文件内容
② 正则解析：[mm:ss.xx]歌词文本
③ 转换为 JSON：[{time: 120.5, text: "歌词"}, ...]
④ 缓存到 Redis cache:lyrics:{songId}（TTL 1800s）

前端：
① requestAnimationFrame 动画循环驱动
② 根据 Audio.currentTime 找到当前歌词索引
③ 高亮当前行，提前 2 行滚动容器
④ 支持点击歌词行跳转到对应进度
```

#### 4.4.3 播放模式

| 模式 | 行为 |
|------|------|
| SEQUENCE | 顺序播放，队列末尾播完自动停止 |
| LOOP | 列表循环，播完从队列头重新开始 |
| REPEAT_ONE | 单曲循环 |
| SHUFFLE | 随机播放，已播放标记防止重复 |

#### 4.4.4 进度记忆

```
① 前端每 10 秒调用 POST /api/progress 上报 currentTime
② 后端写入 Redis progress:{userId}:{songId}（TTL 7天）
③ 进入歌曲时检查是否有进度记录
④ 有 → 弹窗提示"检测到上次听到 xx:xx，是否续播？"

```

Redis Hash room:{roomId}:

#### 6.3.2 WebSocket 端点

```
# 歌房实时通信
ws://host/ws/room/{roomId}?token={accessToken}
消息类型：CHAT（聊天）、SYSTEM（成员变更）、SYNC（播放同步）、HEARTBEAT（心跳）

# 通知实时推送
ws://host/ws/notify?token={accessToken}
消息类型：USER_NOTIFY（用户通知）、SYSTEM_BROADCAST（系统广播）
```

### 6.4 核心流程

#### 6.4.1 房间生命周期

```
创建 → WAITING → PLAYING → (解散/全员离开) → 销毁
                → PAUSED （暂停后继续）
```

- 房间状态和成员信息全部存储在 Redis 中
- `room_id` 使用创建时间的时间戳
- 创建房间需要 VIP 或 ADMIN 角色

#### 6.4.2 播放同步

```
房主操作 → 前端发送 WS type=SYNC → 服务端更新 Redis Hash
         → Redis Pub/Sub 广播 → 房间内所有成员收到 SYNC 消息
         → 成员前端同步 Audio 状态
```

#### 6.4.3 文字聊天

```
用户发送 WS type=CHAT → 服务端校验（≤500字符，频率限制）
         → 广播给房间所有成员
         → 异步写入 MySQL room_message 表（保留最近 50 条）
```

#### 6.4.4 语音通讯

```
按住录音 → MediaRecorder API → 生成 Blob → POST /api/files/upload
         → 上传成功 → 发送 WS type=VOICE → 广播给所有成员
         → 成员前端创建 Audio 播放
```

### 6.5 边界条件

| 边界 | 处理方式 |
|------|---------|
| 房间已满人 | 返回 3002(ROOM_FULL) |
| 私密房间密码错误 | 返回 3004(ROOM_PASSWORD_WRONG) |
| 房主掉线 | 保留房主身份，不断线则保留 |
| 房间全员离开 | 自动解散，清理 Redis |
| 非房主踢人 | 返回 3003(ROOM_NOT_OWNER) |
| 已加入房间再加入 | 返回 3005(ROOM_ALREADY_JOINED) |
| 非 VIP 创建房间 | 返回 3006(ROOM_PERMISSION_DENIED) |

## 10. 文件模块

### 10.1 功能概述

统一文件上传接口，支持音频/图片/LRC/语音，MIME + 魔数双重校验。

### 10.2 文件类型

| type | 说明 | 格式要求 | 大小限制 | 校验方式 |
|------|------|----------|----------|----------|
| audio | 音频 | MP3/AAC | ≤50MB | MIME + 魔数 |
| cover | 封面图 | JPG/PNG | ≤5MB | MIME + 魔数 |
| avatar | 头像 | JPG/PNG | ≤5MB | MIME + 魔数 |
| background | 背景图 | JPG/PNG | ≤5MB | MIME + 魔数 |
| lyric | 歌词 | LRC/TXT | ≤100KB | MIME |
| voice | 语音 | WebM/OGG/WAV | ≤5MB | MIME |

### 10.3 接口

| 方法 | URL | 参数 | 说明 |
|------|-----|------|------|
| POST | /api/files/upload | file, type, roomId(可选) | 通用文件上传 |

> 音频文件不能通过通用上传接口提交，必须走 `/api/members/upload-music` 会员上传流程（含元数据）。

### 10.4 文件存储

```
存储根目录：{project_dir}/data/music/
访问映射：/api/files/{type}/{yyyy}/{MM}/{uuid}.{ext}
存储规则：{type}/{yyyy}/{MM}/{uuid}.{ext}
  音频:   audio/{yyyy}/{MM}/{uuid}.mp3
  封面:   cover/{yyyy}/{MM}/{uuid}.jpg
  歌词:   lyric/{yyyy}/{MM}/{uuid}.lrc
  头像:   avatar/{yyyy}/{MM}/{uuid}.jpg
  背景:   background/{yyyy}/{MM}/{uuid}.jpg
  语音:   voice/{roomId}/{uuid}.webm
```

## 11. 排行榜模块

### 11.1 功能概述

排行榜按歌曲总播放量降序排列，日榜/周榜/月榜/总榜均从数据库 song 表查询（ORDER BY play_count DESC），数据由 PlayHistoryService 在每次播放时实时更新。

### 11.2 接口

| 方法 | URL | 参数 | 说明 |
|------|-----|------|------|
| GET | /api/rankings | type=daily\|weekly\|monthly\|all, limit(默认50) | 获取排行榜（四种类型均按 play_count 降序） |

### 11.3 核心流程

```
各类型榜单统一执行：
  SELECT * FROM song WHERE status = 'ACTIVE'
  ORDER BY play_count DESC LIMIT #{limit}

缓存策略：热门歌曲结果缓存 300s（cache:hot_songs）
```

## 12. 支付模块

### 12.1 功能概述

支付宝沙箱支付集成，支持会员套餐购买。当前为沙箱环境模拟支付。

### 12.2 套餐定义

| 套餐 | 价格 | 时长 |
|------|------|------|
| MONTHLY | ¥15 | 30 天 |
| QUARTERLY | ¥40 | 90 天 |
| YEARLY | ¥128 | 365 天 |

### 12.3 接口方法

| 方法 | URL | 说明 | 鉴权 |
|------|-----|------|------|
| POST | /api/users/me/membership | 会员充值 | JWT |
| POST | /api/payment/alipay/create | 创建支付宝订单 | JWT |
| POST | /api/payment/alipay/notify | 支付宝异步通知 | 公开 |
| GET | /api/payment/alipay/return | 支付宝同步跳转 | 公开 |
| GET | /api/payment/alipay/pay-page | 支付页面 | Token参数 |

### 12.4 核心流程

```
① 用户选择套餐 → 请求创建订单
② 后端生成订单号 → 调用支付宝沙箱 API 获取支付链接
③ 前端跳转到支付宝收银台（沙箱）
④ 用户支付完成 → 支付宝异步通知 notify-url
⑤ 后端验证签名 → 更新订单状态 + 用户会员信息
⑥ 页面重定向回前端
```

## 13. 安全与审计模块

### 13.1 接口限流 @RateLimit

```java
@RateLimit(count = 60, time = 60)  // 60 秒内最多 60 次
```

- 基于 Redis + Lua 脚本实现滑动窗口计数
- 默认 key 为 `ratelimit:{userId}:{uri}`
- 超过限制返回 429 + 错误码 3001

**限流配置**：

| 接口 | 限制 |
|------|------|
| /api/auth/register | 5/min/IP |
| /api/auth/login | 10/min/IP |
| /api/auth/forgot-password | 3/300s |
| /api/songs/{id}/play | 60/min/用户 |

### 13.2 XSS 过滤

所有用户输入（评论/昵称/简介）存储前进行 HTML 转义。

### 13.3 操作审计 @OperationLog

```
@OperationLog(value = "封禁/解封用户", targetType = "USER")
```

- 自定义注解 + AOP 环绕通知
- 自动记录：操作人、操作类型、目标对象/ID、请求参数、响应结果、IP 地址
- 异步写入（@Async），不阻塞主流程

### 13.4 密码安全

- 密码使用 BCrypt 加密存储（strength=4，单次校验 ~50ms）
- 登录支持 RSA 加密传输（前端 jsencrypt 加密，后端解密）
- 密码重置使用 6 位验证码（Redis 5min TTL）

### 13.5 定时任务

当前系统无定时任务。播放量统计已改为实时递增（由 PlayHistoryService 在每次播放时同步更新 song.play_count），不再需要定时批量刷入。

## 14. API 状态码速查

### 业务码规则

| 范围 | 模块 | 示例 |
|------|------|------|
| 200 | 通用成功 | 操作成功 |
| 1xxx | 用户模块 | 1001 邮箱已注册, 1002 账号或密码错误, 1004 账号被封禁 |
| 2xxx | 音乐模块 | 2001 歌曲不存在, 2005 歌曲已存在, 2006 歌单已满 |
| 3xxx | 歌房模块 | 3001 房间不存在, 3003 不是房主, 3006 仅会员可创建歌房 |
| 4xxx | 文件模块 | 4001 文件类型不支持, 4002 文件过大 |
| 5xxx | 会员模块 | 5001 会员过期, 5002 需会员, 5003 上传次数超限, 5004 存储空间不足 |
| 6xxx | 通知模块 | 6001 通知不存在 |
| 7xxx | 举报模块 | 7001 重复举报, 7002 过于频繁 |
| 8xxx | 管理员模块 | 8001 不能封禁自己, 8006 举报已被处理 |
| 9xxx | 通用错误 | 400 参数错误, 401 未登录, 403 无权, 404 不存在, 429 限流 |

### HTTP 状态码

| 状态码 | 说明 | 场景 |
|--------|------|------|
| 200 | 正常响应 | 所有业务码都在 200 响应体中 |
| 401 | 未认证 | Token 缺失/过期/黑名单 |
| 403 | 无权限 | 非管理员访问管理接口 |
| 404 | 资源不存在 | 路由 404、静态资源 404 |
| 429 | 请求过多 | 被限流拦截 |
| 500 | 系统异常 | 数据库宕机、空指针等未预期错误 |

### 统一响应格式

```json
{
    "code": 200,
    "message": "操作成功",
    "data": { ... },
    "timestamp": 1698765432100
}
```
